# FedSDA: Statistical Drift-aware Assignment for Federated Learning — 実装用仕様書

論文「統計的検出によるローカルなコンセプトドリフトに対応可能な連合学習法」(main_jp.tex) を実験プログラム実装用に再構成したもの。

- **ベース手法**: FedDrift (Jothimurugesan et al., 2023) — 複数グローバルモデル + ドリフト検知 + 階層的クラスタリング
- **提案の核**: FedDrift の固定閾値・バッチ単位のヒューリスティックなドリフト検知を、**ADWIN (Bifet & Gavaldà, 2007) による統計的検知**に置き換え、さらに **FIFO 遅延バッファ**により検知遅延中のデータ混入(コンセプト混合)を防ぐ。

---

## 1. 背景(簡潔に)

- 連合学習 (FedAvg) は単一グローバルモデルを学習する枠組みで、時間変化するデータ分布(コンセプトドリフト)を扱えない。
- FedDrift は複数グローバルモデルを保持し、ローカルドリフト検知で新規モデルを生成、階層的クラスタリングで類似モデルを統合する。ただし検知は「損失の増分が固定閾値を超えたか」というヒューリスティックで、統計的保証がない。また固定サイズのバッチ単位で検知するため、バッチ途中でドリフトが起きると新旧コンセプトがバッチ内に混在する。
- 提案手法はデータを **1 件ずつ** 処理し、ADWIN の信頼度パラメータ δ_adwin に基づく統計的判定でノイズと真のドリフトを区別する。検知遅延の問題は FIFO バッファ + 事後分割で解決する。

**貢献**:
1. ADWIN によるローカルドリフト検出機構の導入(検出精度の改善、統計的根拠の付与)
2. FedDrift と比較してドリフト適応速度・モデル性能で優れることを実験的に示す

## 2. 問題設定

### 2.1 システム構成

- クライアント集合 `C = {1, ..., C}`(C 個)+ 中央サーバ 1 台
- 各時刻 t = 1, 2, ... に、各クライアント c は特徴データ x_c^(t) を受け取り、予測 ŷ_c^(t) を出力。その後真ラベル y_c^(t) が判明し、損失 ℓ(ŷ_c^(t), y_c^(t)) を受ける(オンライン学習/テスト・ゼン・トレイン方式)
- データ (x_c^(t), y_c^(t)) は分布 D_c^(t) から生成される
  - **時間方向のドリフト**: D_c^(t) ≠ D_c^(t+1) となり得る
  - **クライアント間の異質性**: c ≠ c' で D_c^(t) ≠ D_c'^(t) となり得る

### 2.2 仮定

1. **損失の有界性**: 損失 ℓ(h(x), y) ∈ [0, 1](必要なら正規化)。Bernstein の不等式による理論保証(ADWIN の統計的閾値)のための条件
2. **データの独立性**: 各 (x_c^(t), y_c^(t)) は D_c^(t) から独立にサンプリング

### 2.3 モデル管理と目的関数

- サーバは通信ラウンド r ごとにグローバルモデル集合 `H^[r] = {h_1, ..., h_|H^[r]|}` を保持
- 各クライアント c はラウンド開始時に H^[r] をダウンロードし、ローカルモデル集合(ワーキング集合)を初期化: `H_c^(t) = H^[r]`
- m_c^(t) = クライアント c が時刻 t に選択しているモデル ID、h_{m_c^(t)} ∈ H_c^(t)

**目的**: 全期間・全クライアントの期待損失総和の最小化

```
min_{(H_c^(t)), (m^(t))}  Σ_{t=1}^{T} Σ_{c∈C}  E_{(x,y)~D_c^(t)} [ ℓ(h_{m_c^(t)}(x), y) ]
```

分布とその変化点は未知なので、各クライアントはストリームから自律的に m_c^(t) を決定し、必要なら新規モデルをローカルで作成する。

## 3. 全体フロー(通信ラウンド r の処理)

論文のシーケンス図 (Fig. 1) に基づく。**サーバ側は FedDrift とほぼ同一**(論文では図のみの簡略説明)。

1 ラウンド = クライアントあたり K ローカルステップ(= K 個のデータ到着)。

```
サーバ                                      クライアント c
  │ ── H^[r] をブロードキャスト ──────────────▶ │
  │                                            │ ワーキング集合初期化: H_c,work^(t0) ← H^[r]
  │                                            │
  │                                            │ ┌─ loop K 回(各時刻 t)──────────────┐
  │                                            │ │ データ (x_c^(t), y_c^(t)) 到着       │
  │                                            │ │ 損失計算・ADWIN でドリフト検知       │
  │                                            │ │ 最適モデル選択 / 新規作成            │
  │                                            │ │ ローカル更新(τ ステップごと)       │
  │                                            │ └────────────────────────────────────┘
  │                                            │
  │                                            │ 最終状態 H_c,work^(rK) から抽出:
  │                                            │   更新済み既存モデル H_c,upd^[r]
  │                                            │   新規モデル H_c,new^[r]
  │ ◀── H_c,upd^[r], H_c,new^[r] をアップロード ─│
  │                                            │
  │ 【FedAvg による集約】                       │
  │   H_upd^[r] ← FedAvg(H_c,upd^[r])          │   ※ モデル ID ごとにクライアント横断で加重平均
  │   H_new^[r] ← ∪_c H_c,new^[r]              │   ※ 新規モデルは全クライアント分を集める
  │                                            │
  │ ── 評価依頼: H_upd^[r] ∪ H_new^[r] ───────▶ │
  │                                            │ 受信した全モデルをローカルデータで評価
  │ ◀── 評価結果(モデルごとの損失等)─────────│
  │                                            │
  │ 【クラスタリング・統合】(FedDrift 準拠)   │
  │   評価値を集計し距離行列を構成             │
  │   階層的クラスタリングで類似モデルをマージ │
  │   → 次ラウンドの H^[r+1] を構築            │
```

### サーバ側の詳細(FedDrift 準拠 — 実装時は原論文 [Jothimurugesan+ 2023] を参照)

- **加重平均**: 同一モデル ID m について、そのモデルを更新した(データストアが非空の)クライアントたちのパラメータを、対応データ量で加重平均(モデルごとの FedAvg)
- **ベースライン損失の集約**: 各モデル m の平均損失をクライアントから集約し、`L̄_base^(m)`(過去全期間の平均損失)を計算して次ラウンドのブロードキャストで共有する(§5 の ResolveDrift で使用)
- **クラスタリング・マージ**(FedDrift の階層的クラスタリング):
  - 全クライアントに全モデルの損失評価を依頼して損失行列 L(m, c) を作る
  - モデル間距離を損失ベースで定義し(FedDrift では概ね「モデル i を モデル j のデータで評価した際の損失悪化量」の対称最大値)、距離が閾値以下のモデル対を加重平均でマージ
  - マージ後の集合が次ラウンドの H^[r+1]

> **注**: 本論文はサーバ側を図のみで説明しており、上記の距離定義・マージ判定の細部は FedDrift 原論文の Algorithm に従うこと。

### モデル間距離の federated な評価(設計判断)

距離行列を作るための損失評価は、**生データをサーバへ集めず、評価対象モデルをデータ保持クライアントへ送って現地で評価させ、集約統計量のみを返させる** federated な形で実装する。具体的には:

1. サーバは距離を測りたいモデル $h_i$ を、モデル $h_j$ に紐づくデータを持つクライアントへブロードキャストする。
2. 各クライアントは自分のローカルデータ上で $h_i$ を評価し、**集約統計量 $(n,\ \sum \ell,\ \sum \ell^2)$ のみ**を返す(生データ・生の予測は返さない)。
3. サーバはこれらを集約して $L_{ij}$(モデル $h_i$ をモデル $h_j$ のデータで評価した平均損失)を構成し、距離 $D(i,j)=\max(L_{ij}-L_{ii},\ L_{ji}-L_{jj},\ 0)$ を得る。

**根拠(この設計は標準的かつ論文と整合)**:

- **FedDrift 原論文の本文と一致**: 論文(algo-creation)は距離を「*the server aggregates the loss estimates $L_{ij}$ of model $h_i$ evaluated over a subsample of the data associated with $h_j$*」と記述し、脚注で「*at client $c$, the data … are subsampled*」とクライアント側でのサブサンプリングを明記している。すなわち **loss 推定値を集約する federated な計算**が意図されている。損失ベースのクラスタ割り当ても「*each client tests the global models … and chooses … the best loss*」(algo-training)と、クライアント側評価として記述されている。
- **クラスタ化FLの標準手法と同型**: 「複数の候補モデルをクライアントがローカル評価して割り当て/距離を決める」形式は IFCA \[Ghosh et al., NeurIPS 2020\] や HypCluster \[Mansour et al. 2020\] の中核であり、連合評価(モデル配布 → ローカル評価 → 指標集約)は FedAvg 系フレームワークの基本機能である。
- **公開コードとの差(意図的)**: FedDrift の公開実装は aggregator が全クライアントの生データ (`all_data`) を保持し中央で距離を計算する**シミュレーション近道**を取っている。本実装はそれに倣わず、上記の federated な評価を採用することで、論文が謳うプライバシー保護の趣旨を実際に満たす。

**トレードオフ**: 本方式は評価を一部のクライアント・一部のサンプル(`CROSS_EVAL_MAX_CLIENTS`, `EVAL_MAX_SAMPLES`)にサブサンプリングする**近似**である(公開コードの全データ厳密計算に対して)。距離推定のノイズが大きい場合はこれらを増やして安定させられる(プライバシーは保ったまま、通信・計算コストと引き換え)。返す集約統計量からはモデル性能が一部漏れうるため、より厳しいプライバシー要件では secure aggregation / 差分プライバシーを追加する余地がある。

## 4. クライアントアルゴリズム(提案手法の核心)

### 4.1 データ構造

**ADWIN 検出器 A**(信頼度パラメータ δ_adwin)

| メソッド | 入力 | 出力 | 動作 |
|---|---|---|---|
| `A.insert(ℓ)` | 損失 ℓ | なし | 内部の動的ウィンドウ W の末尾に損失を追加し、ウィンドウを拡張 |
| `A.detect_drift()` | なし | bool | W 内の**全ての分割点**について δ_adwin を用いて統計的にドリフトを検証し、閾値を超える分割が存在したら True |
| `A.get_split()` | なし | データ ID 集合 I_old, I_new | 検知の根拠となった分割点に基づき、古い概念/新しい概念のデータ ID 集合を返す(データ ID = 時刻 t) |
| `A.shrink()` | なし | なし | W から I_old に対応する損失を削除し、ウィンドウを縮小 |

> **実装注**: 標準の ADWIN は「古い側の削除」までしか行わない。`get_split()` で分割点のインデックス集合を外部に返す点が拡張。ウィンドウ内の各損失に到着時刻 t を紐づけて保持する必要がある。
> ADWIN の判定は、ウィンドウ W を W = W0 · W1 と分割する全ての分割点について、2 つのサブウィンドウの平均差 |μ̂_W0 − μ̂_W1| が統計的閾値 ε_cut(δ_adwin, |W0|, |W1|, 分散)を超えるかを逐次確認するもの(損失が [0,1] に有界なので Bernstein 型の閾値が使える)。既存ライブラリ(例: `river` の ADWIN)をベースに分割点取得を追加する実装が現実的。

**FIFO バッファ F**(バッファ長 N_FIFO)

| メソッド | 入力 | 出力 | 動作 |
|---|---|---|---|
| `F.enqueue(x, y)` | データ (x, y) | なし | 末尾に追加 |
| `F.dequeue()` | なし | (x_out, y_out) | 先頭(最古)のデータを取り出して返す |
| `F.clear()` | なし | なし | 全削除 |

> **実装注**: バッファ内の各データも時刻 t を保持し、`get_split()` の I_old / I_new と突き合わせられるようにする。

**その他の内部状態(クライアント c、ラウンドをまたいで保持)**

- `m_c`: 現在選択中のモデル ID
- `A_c`: ADWIN 検出器
- `S_c = {S_{c,m}}_m`: モデル ID ごとのデータストア(そのモデルで学習すべき実データの蓄積)
- `F_c`: FIFO バッファ

### 4.2 パラメータ一覧

| 記号 | 意味 |
|---|---|
| K | 1 ラウンドあたりのローカルステップ数(データ数) |
| τ | ローカル更新間隔(τ ステップごとに学習) |
| L | ローカル更新回数の係数(τ ステップごとに L·τ 回の勾配更新 → 毎ステップ L 回更新と同等の計算量) |
| E_init | 新規モデル作成時の初期学習エポック数 |
| δ_adwin | ADWIN の信頼度パラメータ |
| N_FIFO | FIFO バッファ長(意図的な遅延の長さ) |
| γ_dist | ResolveDrift の距離閾値(既存モデル適合判定) |

### 4.3 ClientProcess(ラウンド r、クライアント c)

```
入力: サーバからのグローバルモデル集合 H^[r]
出力: H_c,upd^[r](更新済み既存モデル), H_c,new^[r](新規モデル)

 1: t0 ← (r−1)·K;  H_c,work^(t0) ← H^[r]          # ワーキング集合初期化
 2: h_{m_c} ← H_c,work^(t0)[m_c]                    # 選択中モデルのパラメータを同期

 3: # --- ローカルステップ ---
 4: for t = t0+1, ..., r·K:
 5:     H_c,work^(t) ← H_c,work^(t−1)
 6:     データストリームから (x_c^(t), y_c^(t)) を取得
 7:     損失 ℓ_c^(t) ← ℓ(h_{m_c}(x_c^(t)), y_c^(t)) を計算
 8:     A_c.insert(ℓ_c^(t));  F_c.enqueue(x_c^(t), y_c^(t))
 9:     if A_c.detect_drift():
10:         I_old, I_new ← A_c.get_split()
11:         F_c,old ← {(x_c^(s), y_c^(s)) ∈ F_c | s ∈ I_old};  S_{c,m_c} ← S_{c,m_c} ∪ F_c,old
12:         F_c,new ← {(x_c^(s), y_c^(s)) ∈ F_c | s ∈ I_new}
13:         m_c ← ResolveDrift(H_c,work^(t), F_c,new)          # ← Algorithm 2
14:         if m_c が新規 ID:
15:             S_{c,m_c} ← F_c,new;  S_c ← S_c ∪ {S_{c,m_c}}
16:             既存モデルのパラメータから h_{m_c} を初期化
17:             h_{m_c} に対し F_c,new で E_init エポックの初期学習
18:             H_c,work^(t) ← H_c,work^(t) ∪ {h_{m_c}}
19:         else:
20:             S_{c,m_c} ← S_{c,m_c} ∪ F_c,new
21:         A_c.shrink();  F_c.clear()
22:     else if |F_c| > N_FIFO:                                 # 平時: バッファ溢れ分のみ格納
23:         (x_out, y_out) ← F_c.dequeue()
24:         S_{c,m_c} ← S_{c,m_c} ∪ {(x_out, y_out)}
25:     if t mod τ == 0:                                        # 更新間隔ごとに学習
26:         for each m with |S_{c,m}| > 0:
27:             S_{c,m} からミニバッチをサンプリングし h_m に L·τ 回のローカル更新
28:             H_c,work^(t) 内の h_m のパラメータを更新後の状態で上書き

29: # --- サーバ送信用の抽出 ---
30: I^[r] ← { m | h_m ∈ H^[r] }                                # ラウンド開始時の既存モデル ID
31: H_c,upd^[r] ← { h_m ∈ H_c,work^(rK) | m ∈ I^[r] かつ |S_{c,m}| > 0 }
32: H_c,new^[r] ← { h_m ∈ H_c,work^(rK) | m ∉ I^[r] }
33: return H_c,upd^[r], H_c,new^[r]
```

**動作のポイント**:

- **1 件ずつの処理**: 各時刻に 1 データを取得し、損失を ADWIN へ、実データを FIFO バッファへ入れる(8 行目)。バッチ単位の FedDrift と異なり、細かい粒度でドリフトを検知できる
- **ドリフト検知時**(9–21 行目): ADWIN の分割点で FIFO バッファ内の実データを新旧に事後分割。旧データは直前まで使っていたモデルのストアへ、新データは ResolveDrift で決まった切替先モデルのストアへ。最後に ADWIN ウィンドウを縮小しバッファを空にして次の概念に備える
- **平時**(22–24 行目): バッファ長が N_FIFO を超えたときのみ、最古のデータを現在のモデルのストアへ移す。つまりデータは必ず N_FIFO ステップの「保留期間」を経てからストアに確定する — これが検知遅延中の新概念データが旧ストアへ混入するのを防ぐ
- **学習**(25–28 行目): τ ステップごとに、非空の全データストアについてミニバッチ学習(L·τ 回更新)。選択中モデル以外もデータがあれば更新される
- **送信**(29–33 行目): 既存モデルは「実際にローカルデータで更新したもの」のみ、新規モデルは全て送信

### 4.4 設計上の意図(実装時に崩してはいけない性質)

1. **細粒度 + 統計的検出によるコンセプト混合の防止**: FedDrift の固定バッチ検知ではバッチ途中のドリフトで新旧概念が混在する。1 件ずつ処理 + δ_adwin による統計的判定で、粒度と精度(ノイズ耐性)を両立
2. **ADWIN + 遅延バッファの統合(本手法の新規性)**: ドリフト発生から ADWIN 検知までの遅延 (detection delay) 期間中のデータを即座にストアへ入れず FIFO で保留し、検知時に ADWIN の分割点で事後的に振り分ける。これによりデータストアの「純度」を保ち、新規モデルには新概念のデータのみを供給できる

## 5. ResolveDrift(ドリフト解決サブルーチン)

ドリフト検知直後に呼ばれ、新概念データ F_c,new に基づいて「既存モデルへの回帰」か「新規モデル作成」かを決める。

```
入力: 現在のワーキング集合 H_c,work^(t), 新概念データ群 F_c,new
パラメータ: 距離閾値 γ_dist
出力: 新しいモデル ID m*

 1: M_cand ← ∅
 2: for each h_m ∈ H_c,work^(t):
 3:     L̄_new^(m) ← (1/|F_c,new|) Σ_{(x,y)∈F_c,new} ℓ(h_m(x), y)   # 新概念データでの平均損失
 4:     L̄_base^(m) を取得                                            # ベースライン損失(下記)
 5:     if 0 < L̄_base^(m) and L̄_new^(m) − L̄_base^(m) ≤ γ_dist:
 6:         M_cand ← M_cand ∪ {m}
 7: if M_cand ≠ ∅:
 8:     m* ← argmin_{m ∈ M_cand} L̄_new^(m)      # 既知のコンセプト: 最小損失の既存モデル
 9: else:
10:     m* ← 新規のモデル ID                      # 未知のコンセプト: 新規発行
11: return m*
```

**適合判定条件**: `L̄_new^(m) ≤ L̄_base^(m) + γ_dist`
— モデル m が新概念データに対して、過去の性能(ベースライン)から γ_dist 以上悪化していなければ「そのモデルが知っている概念」とみなす。

**ベースライン損失 L̄_base^(m)**: サーバがモデル更新情報を集約する際、モデルごとに各クライアントの平均損失も集約して事前に計算・共有する値(過去全期間の平均損失)。**クライアント自身が経験していない概念でも、システム全体の性能基準で適合判定できる**のがポイント。`L̄_base^(m) > 0` の条件は「まだ一度も評価されていないモデル(ベースライン未定義)」を候補から除外するガード。

## 6. 実装チェックリスト

- [ ] 損失を [0,1] に正規化(ADWIN の統計的閾値の前提)
- [ ] ADWIN: 分割点のデータ ID(時刻)を返す `get_split()` 拡張、`shrink()` での旧側削除
- [ ] FIFO バッファ: 各データに時刻 t を付与し、ADWIN の分割点と整合させる
- [ ] ドリフト検知時の処理順序: 旧データ振り分け → ResolveDrift → 新データ振り分け → shrink & clear
- [ ] 新規モデル初期化: 既存モデルのパラメータからコピーし、F_c,new で E_init エポック学習
- [ ] τ ごとの学習で L·τ 回更新(計算量を τ=1 の場合と揃えるスケーリング)
- [ ] サーバ: モデル ID ごとの FedAvg(データ量加重)、新規モデルの収集、L̄_base^(m) の集約・配布
- [ ] サーバ: 全モデル × 全クライアントの損失評価 → 距離行列 → 階層的クラスタリングによるマージ(FedDrift 原論文準拠)
- [ ] 内部状態(m_c, A_c, S_c, F_c)はラウンドをまたいで永続

## 7. 評価の観点(論文の主張との対応)

- **比較対象**: FedDrift(必要に応じ FedAvg 等のベースライン)
- **指標**: ドリフトへの適応速度(検知遅延・切替後の回復速度)、モデル精度(全期間・全クライアントの平均損失/精度)
